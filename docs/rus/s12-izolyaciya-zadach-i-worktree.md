# Как работает s12: изоляция задач через git worktree

Учебная сессия **s12** завершает линию командной работы из **s09–s11**: агенты уже умеют быть командой, согласовывать важные действия и сами брать задачи с доски. Теперь каждая задача получает **свою отдельную рабочую директорию** через `git worktree`, чтобы параллельная работа не смешивала изменения.

Ориентиры в репозитории:

- код: `agents/s12_worktree_task_isolation.py`;
- англоязычный первоисточник: `docs/en/s12-worktree-task-isolation.md`.

Путь в курсе: `s01 > s02 > s03 > s04 > s05 > s06 | s07 > s08 > s09 > s10 > s11 > [ s12 ]`.

> *«Каждый работает в своей директории и не мешает другим»* — задачи управляют целями, worktree управляют рабочими папками, связь держится через номер задачи.

---

## Суть: зачем эта глава

В **s11** агенты уже умеют сами брать задачи с доски. Но все задачи всё ещё выполняются в одной общей папке проекта. Если два агента одновременно меняют разные части системы, их незавершённые изменения могут оказаться в одном рабочем дереве Git.

Представьте:

```text
Одна общая папка проекта

┌─────────────────────────────┐
│ learn-claude-code/          │
│                             │
│ agent A меняет config.py    │
│ agent B тоже меняет config.py
│ agent C запускает тесты     │
└─────────────────────────────┘
```

Что может пойти не так:

1. **Изменения смешиваются.** Один агент правит `config.py` для задачи про авторизацию, второй — для задачи про интерфейс. В рабочем дереве Git всё оказывается в одной куче.
2. **Непонятно, чья правка к какой задаче относится.** Доска задач знает, *что нужно сделать*, но не знает, *где именно выполнялась работа*.
3. **Сложно откатить только одну задачу.** Если изменения нескольких задач лежат в одной папке, нельзя легко сказать: «удалить только попытку по задаче 1, но сохранить задачу 2».
4. **Параллельная работа становится опасной.** Один агент может менять файл, который другой агент уже трогает.

Коротко:

```text
s11 решает вопрос: «КТО взял задачу?»
s12 добавляет ответ: «ГДЕ эта задача выполняется?»
```

## Решение s12

**s12** разделяет две плоскости:

```text
Плоскость управления                 Плоскость выполнения

.tasks/                              .worktrees/
┌──────────────────┐                 ┌─────────────────────┐
│ task_1.json      │                 │ auth-refactor/       │
│ status: in_progress ◄────────────► │ отдельная папка      │
│ worktree: auth-refactor            │ branch: wt/auth-...  │
└──────────────────┘                 └─────────────────────┘

┌──────────────────┐                 ┌─────────────────────┐
│ task_2.json      │                 │ ui-login/            │
│ status: pending  ◄────────────────►│ отдельная папка      │
│ worktree: ui-login                 │ branch: wt/ui-login  │
└──────────────────┘                 └─────────────────────┘
```

**`.tasks/`** отвечает за цель: какая задача, какой статус, кто владелец, к какой рабочей папке привязана.

**`.worktrees/`** отвечает за место выполнения: отдельная директория, отдельная ветка, отдельное Git-состояние.

Связка между ними — **номер задачи** (`task_id`) и имя рабочей папки.

---

## Что такое git worktree простыми словами

Обычно у репозитория есть одна рабочая папка: вы редактируете файлы, Git видит изменения.

`git worktree` позволяет создать **ещё одну рабочую папку того же репозитория**, но с отдельной веткой.

Примерно так:

```text
Основная папка:
learn-claude-code/

Дополнительная рабочая папка:
learn-claude-code/.worktrees/auth-refactor/

Внутри — те же файлы проекта,
но изменения идут в отдельной ветке wt/auth-refactor.
```

Это удобно для агентов: один агент может работать в `auth-refactor`, другой — в `ui-login`, и их незакоммиченные изменения не будут лежать в одной общей папке.

## Как это устроено в s12

### 1. Сначала создаётся задача

Задача сохраняется в `.tasks/`:

```text
.tasks/task_1.json

id: 1
subject: Implement auth refactor
status: pending
worktree: ""
```

На этом этапе есть цель, но ещё нет отдельного места выполнения.

### 2. Затем создаётся worktree и привязывается к задаче

Когда вызывается создание рабочей папки для задачи, происходит несколько действий:

```text
task_1.json                       .worktrees/auth-refactor/
status: in_progress               branch: wt/auth-refactor
worktree: auth-refactor            task_id: 1
```

Если задача была `pending`, она автоматически переходит в `in_progress`.

### 3. Команды выполняются внутри worktree

Инструмент `worktree_run` запускает команду не в основной папке, а в выбранной рабочей директории.

То есть не просто:

```text
запусти git status где угодно
```

а:

```text
запусти git status внутри .worktrees/auth-refactor/
```

Это и даёт изоляцию.

### 4. В конце worktree можно сохранить или удалить

Есть два разных сценария завершения:

- **`worktree_keep`** — оставить папку для дальнейшей проверки;
- **`worktree_remove`** — удалить папку; при `complete_task=true` ещё и отметить связанную задачу как `completed`.

Это важное различие: иногда результат нужно сохранить для ревью, а иногда рабочую область можно закрыть.

## Жизненные состояния

В **s12** фактически есть две связанные машины состояний.

```text
Задача:
pending ──► in_progress ──► completed

Рабочая папка:
absent ──► active ──► kept
                  └──► removed
```

Задача отвечает на вопрос: **что делаем и в каком статусе цель**.

Worktree отвечает на вопрос: **где выполнялась работа и что стало с рабочей папкой**.

---

## Журнал событий

В **s12** появляется журнал жизненного цикла:

```text
.worktrees/events.jsonl
```

Туда пишутся события:

- `worktree.create.before`;
- `worktree.create.after`;
- `worktree.create.failed`;
- `worktree.remove.before`;
- `worktree.remove.after`;
- `worktree.remove.failed`;
- `worktree.keep`;
- `task.completed`.

Зачем это нужно?

Если чат с моделью забыт, сжат или процесс упал, состояние можно восстановить по файлам:

```text
.tasks/                  что было с задачами
.worktrees/index.json    какие рабочие папки создавались
.worktrees/events.jsonl  что происходило по шагам
```

Ключевая мысль: **память разговора ненадёжна, состояние на диске долговечнее**.

## Что изменилось по сравнению с s11

| Компонент | s11 | s12 |
| --- | --- | --- |
| Координация | Доска задач: владелец и статус | Доска задач + явная привязка к worktree |
| Где выполняется работа | Общая папка проекта | Отдельная папка под задачу |
| Восстановление после сбоя | В основном статус задачи | Статус задачи + индекс worktree + журнал событий |
| Завершение | Отметить задачу | Решить: оставить worktree или удалить |
| Видимость процесса | По логам и файлам задач | Явные события в `.worktrees/events.jsonl` |

## Как запустить

Из корня репозитория:

```sh
python agents/s12_worktree_task_isolation.py
```

Важно: инструменты `worktree_*` требуют, чтобы запуск происходил внутри Git-репозитория. Если Git-репозиторий не найден, скрипт напечатает предупреждение, а worktree-инструменты будут возвращать ошибки.

Что попробовать:

1. Создать задачи для backend auth и frontend login page, затем вывести список задач.
2. Создать worktree `auth-refactor` для задачи 1.
3. Создать worktree `ui-login` для задачи 2.
4. Выполнить `git status --short` внутри `auth-refactor`.
5. Оставить `ui-login` через `worktree_keep`.
6. Удалить `auth-refactor` через `worktree_remove` с `complete_task=true`.
7. Посмотреть задачи, список worktree и журнал событий.

## FAQ

*Для тех, кто впервые сталкивается с LLM, агентами, Git worktree и изоляцией параллельных задач.*

### Зачем нужен worktree, если уже есть доска задач?

Доска задач отвечает на вопрос **«что делать?»**. Worktree отвечает на вопрос **«где это делать?»**.

Без worktree две задачи могут смешать изменения в одной папке. С worktree каждая задача получает отдельную рабочую область.

### Это то же самое, что отдельная копия репозитория?

Похоже по ощущению, но технически это механизм Git. `git worktree` создаёт дополнительную рабочую директорию, связанную с тем же репозиторием, но обычно с отдельной веткой. Это легче и аккуратнее, чем вручную копировать весь проект.

### Почему нельзя просто создать папку `task-1/` и скопировать туда файлы?

Можно, но тогда Git не будет нормально понимать связь этой папки с репозиторием, ветками и статусом изменений. Worktree сохраняет Git-механику: ветки, статус, историю, команды `git status`, `git diff`.

### Что значит «задача привязана к worktree»?

Это значит, что в файле задачи записано имя рабочей папки, например:

```text
task_1.json → worktree: auth-refactor
```

А в индексе worktree есть обратная информация:

```text
auth-refactor → task_id: 1
```

Так можно понять: задача 1 выполнялась в рабочей папке `auth-refactor`.

### Зачем нужен `index.json`, если Git и так знает worktree?

Git знает про worktree на своём уровне, но учебному агенту нужен простой файл состояния, который легко читать инструментами. `index.json` показывает, какие worktree отслеживает именно эта агентская система, с какими задачами они связаны и в каком они статусе: `active`, `kept`, `removed`.

### Что значит `keep`?

`keep` означает: «не удалять рабочую папку, оставить её для дальнейшей проверки». Например, агент закончил черновик, но человек хочет посмотреть файлы или прогнать дополнительные проверки.

### Что значит `remove` с `complete_task=true`?

Это означает: «удалить рабочую папку и считать связанную задачу завершённой». Код дополнительно обновляет задачу в `.tasks/`, снимает привязку к worktree и пишет событие `task.completed`.

### Можно ли потерять работу при `remove`?

Да, если удалить worktree с незасохранёнными важными изменениями. Поэтому в реальных проектах перед удалением нужно проверять статус, сохранять нужные изменения, делать коммит или явно принимать решение, что рабочую папку можно удалить.

Учебная глава показывает механизм, но не заменяет осторожность при работе с Git.

### Зачем нужен журнал событий?

Он помогает ответить на вопросы:

- когда создали рабочую папку;
- с какой задачей она была связана;
- удалили её или оставили;
- была ли задача отмечена завершённой;
- где произошла ошибка.

Это полезно после сбоя, при отладке и при разборе действий агента.

### Чем s12 завершает линию s09–s12?

- **s09:** появилась команда агентов и почта.
- **s10:** появились правила важных решений.
- **s11:** участники начали сами брать задачи.
- **s12:** каждая задача получила отдельное место выполнения.

То есть система постепенно идёт от «один агент делает всё в одной папке» к модели, где несколько участников могут параллельно работать над разными задачами с отдельными рабочими областями.
