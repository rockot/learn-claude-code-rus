# s05: Skill Loading — полное руководство

Объединённый документ: как работает загрузка скиллов, как агент по ним работает и как модель решает, когда скилл нужен.

---

## Часть 1: Как работает s05 Skill Loading

s05 добавляет **навыки** — специализированные инструкции, которые модель загружает по запросу, а не получает целиком в системном промпте.

### Проблема

Нужно дать агенту доменные знания (git, тесты, code review). Если всё класть в системный промпт:
- 10 навыков по ~2000 токенов ≈ 20 000 токенов;
- большинство не используется в конкретной задаче;
- лишние токены тратятся на каждый запрос.

### Решение: два уровня

| Уровень | Где | Что | Токены |
|---------|-----|-----|--------|
| **Layer 1** | Системный промпт | Только имена и краткие описания | ~100 на навык |
| **Layer 2** | tool_result | Полный текст навыка | По запросу |

Модель видит список навыков и вызывает `load_skill(name)`, когда нужен конкретный навык. Полный текст попадает в контекст только после вызова.

### Структура навыков

```
skills/
  pdf/
    SKILL.md          # ---\n name: pdf\n description: ...\n ---\n <тело>
  code-review/
    SKILL.md
  agent-builder/
    SKILL.md
```

Каждый навык — файл `SKILL.md` с YAML frontmatter и телом.

### SkillLoader

- **Layer 1** (`get_descriptions`) — краткие описания для системного промпта.
- **Layer 2** (`get_content`) — полный текст в формате `<skill name="...">...</skill>` для tool_result.

### Инструмент load_skill

При вызове `load_skill(name="code-review")` в `tool_result` возвращается полный текст SKILL.md. Он добавляется в `messages` и становится доступен модели в следующем вызове.

### Итог

- **Layer 1** — имена и описания в системном промпте (~100 токенов на навык).
- **Layer 2** — полный текст через `load_skill` в tool_result.
- **Принцип** — загружать знания по запросу, а не заранее в системный промпт.

## Часть 2: Как агент работает по скиллу

Скилл — это текстовая инструкция. Модель **читает** его и **действует** по нему, как по обычной инструкции в контексте.

### Загрузка скилла

```
Модель вызывает: load_skill(name="code-review")
         ↓
tool_result: "<skill name=\"code-review\">\n# Code Review\n\nCheck for:\n- ...\n</skill>"
         ↓
Этот текст добавляется в messages как часть следующего user-сообщения
```

Скилл оказывается в `messages` как обычный текст, который модель видит в следующем запросе.

### Как модель его использует

Модель получает:
- системный промпт;
- историю диалога (включая только что загруженный скилл);
- список инструментов.

Она **интерпретирует** текст скилла и **решает**, что делать дальше: какие инструменты вызвать и в каком порядке.

Пример для `code-review`:

```
Скилл: "Check for: code quality, style consistency, test coverage..."
         ↓
Модель: "Нужно прочитать файлы, проверить по чек-листу"
         ↓
read_file("hello.py")
         ↓
Скилл: "Style: PEP8, type hints..."
         ↓
Модель: "Нет type hints, добавляю"
         ↓
edit_file(...)
```

### Нет «выполнения» скилла

Скилл — это текст, а не код. Его нет:
- никакого парсера шагов;
- никакого выполнения по формату;
- никакой автоматической проверки.

Модель **сама** читает инструкцию и решает, что делать. Она не обязана следовать скиллу строго, но обучена следовать инструкциям в контексте.

### Роль скилла

**Без скилла:** модель опирается на общие знания.

**Со скиллом:** модель получает точные шаги, чек-листы и правила для конкретной области.

Скилл задаёт **контекст** и **формат** работы, а не механизм выполнения.

## Часть 3: Как модель понимает, что нужен скилл

Модель **не знает** заранее, когда нужен скилл. Она решает это по **системному промпту** и **запросу пользователя**.

### Что модель видит

**Системный промпт:**

```
You are a coding agent at {WORKDIR}.
Use load_skill to access specialized knowledge before tackling unfamiliar topics.

Skills available:
  - agent-builder: Design and build AI agents...
  - code-review: Review code for quality and style
  - pdf: Process PDF files
  - mcp-builder: Build MCP servers
```

Модель получает:
- инструкцию: «используй load_skill для доступа к знаниям»;
- список навыков и их краткие описания.

### Как модель выбирает скилл

Она сопоставляет **запрос пользователя** с **описаниями** навыков.

| Запрос пользователя | Описание скилла | Решение модели |
|--------------------|-----------------|----------------|
| "Do a code review" | "Review code for quality and style" | Вызвать `load_skill("code-review")` |
| "Build an MCP server" | "Build MCP servers" | Вызвать `load_skill("mcp-builder")` |
| "Process this PDF" | "Process PDF files" | Вызвать `load_skill("pdf")` |
| "Create an agent" | "Design and build AI agents" | Вызвать `load_skill("agent-builder")` |

Это тот же механизм, что и для других инструментов: модель выбирает по `description` и контексту.

### Что значит «unfamiliar» — незнакомая тема

LLM обучена на большом объёме текста и имеет общие знания по многим темам. В строгом смысле для неё почти всё «знакомо».

«Unfamiliar» в системном промпте — это не про пробелы в знаниях, а подсказка для поведения модели. По сути это означает:

- **Когда лучше загрузить скилл**, а не полагаться только на общие знания
- **Когда задача требует специфичного подхода**, а не «как модель обычно делает»

Зачем скилл, если модель «всё знает»:

| Аспект | Общие знания модели | Скилл |
|--------|---------------------|--------|
| **Глубина** | Общие практики | Конкретные шаги, чек-листы |
| **Согласованность** | Может варьироваться | Один и тот же процесс |
| **Политика** | Не знает ваших правил | Ваши правила и ограничения |
| **Контекст** | Общий | Под вашу область/проект |

«Unfamiliar» — это «если задача требует специализированного подхода — используй load_skill».

## Часть 4: Как модель выбирает — общие знания или скилл

Отдельного механизма «понимания» или «решения» у модели нет. Это тот же процесс предсказания следующего токена.

### Что на самом деле происходит

Модель не «думает» и не «выбирает» в привычном смысле. Она предсказывает наиболее вероятное продолжение текста с учётом:

- системного промпта;
- истории диалога;
- списка инструментов и их описаний.

В этом контексте она генерирует последовательность токенов, которая может включать вызов инструмента.

### Почему она иногда вызывает load_skill

При обучении модель видела много примеров, где:
- в инструкциях говорится «используй X, когда Y»;
- при запросе «сделай X» вызывается инструмент, описание которого совпадает с X.

В результате для контекста вроде:

```
System: Skills available: code-review: Review code for quality...
User: Do a code review of hello.py
```

последовательность, ведущая к вызову `load_skill("code-review")`, получает более высокую вероятность, чем варианты без вызова.

### Роль системного промпта

Фраза «Use load_skill before tackling unfamiliar topics» меняет распределение вероятностей:
- без неё модель чаще полагается на общие знания;
- с ней чаще генерируются вызовы `load_skill`, когда запрос похож на описание скилла.

Это не логика, а влияние текста на вероятности токенов.

### Кратко

- Модель не «понимает» и не «решает» — она предсказывает следующие токены.
- Вызов скилла — один из возможных вариантов продолжения.
- Обучение и контекст (промпт, описания инструментов) задают, насколько вероятен вызов `load_skill`.
- «Понимание» — это следствие того, как обучение связало описания скиллов с типами запросов.

## Часть 5: Tool vs Skill — в чём разница

В терминологии learn-claude-code это разные сущности.

### Tool (инструмент)

- **Что это:** функция, которую агент вызывает через tool_use.
- **Результат:** handler выполняет код и возвращает `tool_result`.
- **Примеры:** `read_file`, `bash`, `write_file`, `load_skill`.
- **Внешний сервис:** standalone-скрипт (например, fireflies_obsidian_service) можно обернуть в tool — handler запускает `subprocess.run(...)` и возвращает вывод. Сервис становится **реализацией** tool.

### Skill (навык)

- **Что это:** текстовая инструкция (SKILL.md), которую агент загружает по запросу.
- **Результат:** текст попадает в `messages`; модель читает и интерпретирует его.
- **Примеры:** `code-review`, `pdf`, `agent-builder`.
- **Не исполняется:** скилл — документация, а не код. Модель сама решает, как действовать по инструкции.

### Сравнение

| Аспект | Tool | Skill |
|--------|------|-------|
| **Суть** | Исполняемый код | Текстовые инструкции |
| **Вызов** | Модель вызывает → handler выполняется | Модель вызывает `load_skill` → получает текст |
| **Результат** | Вывод программы/функции | Текст в контексте |
| **Внешний сервис** | Может быть реализацией tool | Не применимо |

### Пример: fireflies_obsidian_service

Сервис интеграции Fireflies и Obsidian — это **tool** (или реализация tool):
- standalone Python-скрипт с фиксированным пайплайном;
- агент может вызвать tool `sync_fireflies_meeting(meeting_id)`, handler запускает `python main.py --meeting_id X`;
- это **не** subagent (нет цикла, нет tool_use) и **не** skill (это код, а не документация).

## Часть 6: Как модель понимает, что нужен субагент

Механизм тот же, что и для tool/skill: модель опирается на **описание инструмента** и **системный промпт**.

### Что видит модель

**Системный промпт (s04):**
```
You are a coding agent at {WORKDIR}. Use the task tool to delegate exploration or subtasks.
```

**Описание инструмента `task`:**
```
Spawn a subagent with fresh context. It shares the filesystem but not conversation history.
```

### Когда модель вызывает task

Она сопоставляет **запрос пользователя** с формулировками «delegate», «exploration», «subtasks» и описанием инструмента.

| Запрос пользователя | Почему модель может вызвать task |
|--------------------|-----------------------------------|
| "Use a subtask to find what testing framework this project uses" | Прямое указание «subtask» + исследование |
| "Delegate: read all .py files and summarize" | «Delegate» + много чтения |
| "Use a task to create a new module, then verify from here" | «task» + разделение на подзадачу и проверку |

### Что подсказывает вызов task

- **Явные слова** — «subtask», «delegate», «task» в запросе.
- **Исследование** — много чтения файлов, поиск информации.
- **Изоляция контекста** — подзадача с большим числом tool_use, чтобы не засорять родительский контекст.
- **Системный промпт** — «Use the task tool to delegate exploration or subtasks».

### Технически

Как и для tool/skill — это **предсказание следующего токена**. Модель видит системный промпт и описание `task`; при обучении она встречала паттерны «делегирование → вызов инструмента». Для подходящего контекста вызов `task` получает более высокую вероятность.

### Кратко

- Модель не «понимает» необходимость субагента — она генерирует вызов `task` как один из вариантов продолжения.
- Решение основано на системном промпте, описании инструмента и запросе пользователя.
- Чем явнее в запросе «subtask», «delegate», «exploration», тем выше вероятность вызова `task`.

## Попробуй сам

```sh
cd learn-claude-code
python agents/s05_skill_loading.py
```

1. `Какие скиллы доступны?`
2. `Загрузи скилл agent-builder и выполни инструкции из него`
3. `Мне нужно сделать code review — сначала загрузи подходящий скилл`
4. `Собери MCP-сервер с помощью скилла mcp-builder`
