# s07: Task System — полное руководство

Объединённый документ: что это, как работает, CRUD, откуда берутся таски и как модель решает их создавать.

---

## Часть 1: Что это и как работает

s07 вводит **файловый граф задач** с зависимостями. Задачи хранятся на диске и переживают сжатие контекста (s06) и перезапуски.

### Проблема s03 (TodoManager)

- Плоский чек-лист в памяти
- Нет порядка и зависимостей
- Только done/not done
- При сжатии контекста (s06) список теряется

Реальные цели имеют структуру: B зависит от A, C и D можно делать параллельно, E ждёт C и D.

### Решение: граф задач на диске

Каждая задача — JSON-файл в `.tasks/`:

```
.tasks/
  task_1.json  {"id":1, "subject":"Setup project", "status":"completed", "blockedBy":[], "blocks":[2,3]}
  task_2.json  {"id":2, "subject":"Write code", "blockedBy":[1], "status":"pending"}
  task_3.json  {"id":3, "subject":"Write tests", "blockedBy":[1], "status":"pending"}
  task_4.json  {"id":4, "subject":"Integration", "blockedBy":[2,3], "status":"pending"}
```

Граф отвечает на три вопроса:

- **Что готово?** — `pending` и пустой `blockedBy`
- **Что заблокировано?** — `blockedBy` не пуст
- **Что сделано?** — `completed`, снятие блокировок у зависимых

### Структура задачи

| Поле | Тип | Назначение |
|------|-----|------------|
| `id` | int | Уникальный ID |
| `subject` | str | Краткое описание |
| `description` | str | Подробное описание |
| `status` | str | `pending` → `in_progress` → `completed` |
| `blockedBy` | list[int] | ID задач, от которых зависит эта |
| `blocks` | list[int] | ID задач, которые зависят от этой |
| `owner` | str | Исполнитель (для s09+) |

### TaskManager

- **create** — создаёт задачу и сохраняет в `task_{id}.json`
- **update** — меняет статус и зависимости; при `completed` вызывает `_clear_dependency`
- **_clear_dependency** — удаляет завершённую задачу из `blockedBy` у всех остальных
- **list_all** — возвращает список с маркерами `[ ]`, `[>]`, `[x]`

### Инструменты

| Инструмент | Действие |
|------------|----------|
| `task_create` | Создать задачу |
| `task_update` | Обновить статус и зависимости |
| `task_list` | Список всех задач |
| `task_get` | Получить задачу по ID |

### Пример графа

```
                 +----------+
            +--> | task 2   | --+
            |    | pending  |   |
+----------+     +----------+    +--> +----------+
| task 1   |                          | task 4   |
| completed| --> +----------+    +--> | blocked  |
+----------+     | task 3   | --+     +----------+
                 | pending  |
                 +----------+

Порядок: task 1 → task 2 и 3 (параллельно) → task 4
```

### Зачем хранить на диске

- Сжатие контекста (s06) не затрагивает `.tasks/`
- Задачи сохраняются между сессиями
- Граф — основа для s08 (фоновые задачи), s09+ (команды агентов), s12 (worktree)

## Часть 2: Что такое CRUD

**CRUD** — аббревиатура четырёх базовых операций с данными:

| Буква | Операция | Англ. | Пример |
|-------|----------|-------|--------|
| **C** | Создание | Create | `task_create`, `create()` |
| **R** | Чтение | Read | `task_get`, `task_list`, `get()`, `list_all()` |
| **U** | Обновление | Update | `task_update`, `update()` |
| **D** | Удаление | Delete | (в s07 удаления нет) |

В TaskManager (s07) реализованы Create, Read и Update. Delete обычно добавляют отдельно, если нужно удалять задачи.

## Часть 3: Откуда берутся таски и как запрос превращается в задачи

Таски создаёт **модель**, когда вызывает инструменты `task_create` и `task_update`. Автоматического превращения запроса в задачи нет — это делает LLM.

### Цепочка

```
Запрос пользователя
       ↓
Модель читает запрос + системный промпт ("Use task tools to plan and track work")
       ↓
Модель решает создать задачи и вызывает task_create, task_update
       ↓
TaskManager сохраняет в .tasks/task_N.json
```

### Пример

**Пользователь:**  
*"Create 3 tasks: Setup project, Write code, Write tests. Make them depend on each other in order."*

**Модель:**
1. Вызывает `task_create(subject="Setup project")` → создаётся task_1
2. Вызывает `task_create(subject="Write code")` → создаётся task_2
3. Вызывает `task_create(subject="Write tests")` → создаётся task_3
4. Вызывает `task_update(task_id=2, addBlockedBy=[1])` → task_2 зависит от task_1
5. Вызывает `task_update(task_id=3, addBlockedBy=[2])` → task_3 зависит от task_2

**Результат:** цепочка 1 → 2 → 3 в `.tasks/`.

### Кто принимает решения

- **Пользователь** — формулирует цель и иногда явно просит создать задачи.
- **Модель** — решает, когда создавать задачи, как их называть и как связывать зависимостями.
- **Системный промпт** — подсказывает: «Use task tools to plan and track work».

## Часть 4: Когда модель создаёт задачи, а когда нет

Если в запросе пользователя явно не указано создание задач, модель может как принять решение создать их, так и действовать без них.

### На чём основано решение модели

Отдельной логики «понимания» нет. Решение — это **предсказание следующего токена** с учётом:

1. **Системного промпта** — «Use task tools to plan and track work» повышает вероятность использования task tools.
2. **Описаний инструментов** — модель видит `task_create`, `task_update` и их описания.
3. **Запроса пользователя** — сложность и формулировки влияют на выбор.

### Что увеличивает вероятность создания задач

| Признаки запроса | Почему модель чаще создаёт задачи |
|------------------|-----------------------------------|
| Многошаговая задача | «Refactor the project», «Add auth and tests» |
| Слова «сначала», «потом», «шаги» | Явно указывают на последовательность |
| Несколько файлов/модулей | «Update config, utils, and main» |
| Неявная сложность | «Build a full feature» — модель может разбить на шаги |

### Что уменьшает вероятность

| Признаки запроса | Почему модель чаще обходится без задач |
|------------------|----------------------------------------|
| Одна простая операция | «Read hello.py», «Add a comment» |
| Один файл | «Fix the typo in config» |
| Нет явной структуры | «Do something useful» |

### Кратко

Модель не «понимает» необходимость задач — она выбирает вызовы инструментов по вероятностям. Системный промпт и сложность запроса подталкивают к созданию задач, но гарантий нет. Если задача должна всегда планироваться через tasks, это нужно явно указать в промпте.

## Что изменилось по сравнению с s06

| Компонент | Было (s06) | Стало (s07) |
|---|---|---|
| Инструменты | 5 | 8 (`task_create`, `task_update`, `task_list`, `task_get`) |
| Модель планирования | Плоский чек-лист (в памяти) | Граф задач с зависимостями (на диске) |
| Связи между задачами | Нет | `blockedBy` + `blocks` |
| Статусы | Условно done/not done | `pending` -> `in_progress` -> `completed` |
| Персистентность | Теряется при сжатии | Переживает сжатие и перезапуск |
