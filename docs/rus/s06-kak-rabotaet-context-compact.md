# Как работает s06: Context Compact

s06 добавляет **трёхуровневую стратегию сжатия контекста**, чтобы агент мог работать бесконечно долго, не упираясь в лимит окна контекста.

---

## Проблема

Окно контекста ограничено. Один `read_file` на файл в 1000 строк — ~4000 токенов. После 30 файлов и 20 bash-команд — 100 000+ токенов. Без сжатия агент не справляется с большими проектами.

---

## Решение: три уровня

```
Каждый ход:
  Tool results
       ↓
[Layer 1: micro_compact]     ← каждый ход, незаметно
  Старые tool_result → "[Previous: used {tool_name}]"
       ↓
[Токены > 50000?]
   нет        да
    ↓          ↓
 continue   [Layer 2: auto_compact]
              Сохранить транскрипт в .transcripts/
              LLM суммирует диалог
              Заменить messages на [summary]
                    ↓
            [Layer 3: compact tool]
              Модель вызывает compact → то же сжатие по запросу
```

---

## 1. Layer 1: micro_compact (строки 67–93)

**Когда:** перед каждым вызовом LLM.

**Что делает:**
- Находит все `tool_result` в `messages`.
- Оставляет последние `KEEP_RECENT` (3) без изменений.
- Для более старых: если `content` длиннее 100 символов, заменяет на `"[Previous: used {tool_name}]"`.

**Пример:**
```
Было: tool_result с 5000 символов вывода read_file
Стало: "[Previous: used read_file]"
```

**Эффект:** старые большие выводы не занимают контекст, но модель знает, что инструмент вызывался.

---

## 2. Layer 2: auto_compact (строки 97–121)

**Когда:** если `estimate_tokens(messages) > THRESHOLD` (50 000).

**Что делает:**
1. Сохраняет полный транскрипт в `.transcripts/transcript_{timestamp}.jsonl`.
2. Отправляет LLM запрос: «Summarize this conversation for continuity...» + до 80 000 символов диалога.
3. Заменяет весь `messages` на два сообщения:
   - user: `"[Conversation compressed. Transcript: ...]\n\n{summary}"`
   - assistant: `"Understood. I have the context from the summary. Continuing."`

**Эффект:** контекст сильно сокращается, полная история остаётся в файле.

---

## 3. Layer 3: compact tool (строки 176, 189, 213–227)

**Когда:** модель сама вызывает инструмент `compact`.

**Что делает:** тот же алгоритм, что и auto_compact — сохранение транскрипта, суммирование, замена `messages`.

**Зачем:** модель может сжать контекст до того, как сработает порог, например перед сложной задачей.

---

## 4. Порядок в цикле (строки 195–227)

```python
while True:
    micro_compact(messages)                    # Layer 1 — каждый ход
    if estimate_tokens(messages) > THRESHOLD:
        messages[:] = auto_compact(messages)   # Layer 2 — при переполнении
    response = client.messages.create(...)
    # ... выполнение инструментов ...
    if manual_compact:
        messages[:] = auto_compact(messages)   # Layer 3 — после вызова compact
```

---

## 5. Оценка токенов (строки 62–64)

```python
def estimate_tokens(messages: list) -> int:
    return len(str(messages)) // 4  # ~4 символа на токен
```

Грубая оценка, без точного токенизатора.

---

## 6. Транскрипты

Полная история сохраняется в `.transcripts/`. Ничего не теряется — только выносится из активного контекста. При необходимости можно восстановить детали из файлов.

---

## Итог

- **micro_compact** — заменяет старые tool_result на короткие плейсхолдеры каждый ход.
- **auto_compact** — при превышении порога сохраняет транскрипт и заменяет диалог кратким саммари.
- **compact tool** — ручное сжатие по инициативе модели.
- **Транскрипты** — полная история в `.transcripts/` для восстановления.

# Вопросы и пояснения

## 1. Почему выбран порог в 50 000 токенов?

В коде это просто константа:

```python
THRESHOLD = 50000
```

Жёсткого обоснования нет. Скорее всего:

- **Запас до лимита модели** — у Claude Sonnet окно ~200k токенов, 50k оставляет место для ответа и дальнейшей работы.
- **Практический компромисс** — не сжимать слишком рано (чтобы не терять контекст) и не ждать до самого лимита.
- **Удобная величина** — типичный выбор в примерах и туториалах.

Для других моделей (меньше/больше окно) порог лучше подбирать отдельно.

---

## 2. Каким образом поддерживается качество передаваемой информации после каждого сжатия?

**Кратко:** явного контроля нет. Всё держится на промпте для саммаризации.

### Что есть сейчас

Промпт для LLM (строки 110–112):

```python
"Summarize this conversation for continuity. Include: "
"1) What was accomplished, 2) Current state, 3) Key decisions made. "
"Be concise but preserve critical details."
```

То есть:
- задаётся структура саммари (что сделано, текущее состояние, решения);
- просят сохранять «critical details»;
- никакой проверки или валидации результата нет.

### Риски

- LLM может посчитать важные детали (пути, фрагменты кода, имена файлов) «неважными» и опустить их.
- «Be concise» может подталкивать к излишнему сокращению.
- Нет гарантии, что конкретные факты (например, «файл X уже изменён») попадут в саммари.

### Что можно улучшить

| Подход | Описание |
|--------|----------|
| **Более точный промпт** | Явно перечислить, что сохранять: пути, имена файлов, ключевые изменения, ошибки. |
| **Структурированный формат** | Задать шаблон саммари (JSON/YAML) с полями: `files_modified`, `current_task`, `decisions`, `errors`. |
| **Два прохода** | Сначала извлечь факты (файлы, команды), потом написать текст саммари. |
| **Проверка** | Сравнивать саммари с исходным контекстом (ключевые сущности, имена файлов) и при необходимости пересжимать. |
| **Селективное сжатие** | Сжимать только старые части, оставлять последние N сообщений без изменений. |

### Итог

Порог 50k — практический выбор, не строго обоснованный. Качество саммари не контролируется: оно зависит только от промпта и поведения LLM. В s06 это учебный пример; в продакшене нужны более жёсткие правила и, при необходимости, проверки.
