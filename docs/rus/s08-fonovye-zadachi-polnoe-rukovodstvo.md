# Как работает s08: фоновые задачи (background tasks)

Учебная сессия **s08** в репозитории *learn-claude-code* добавляет к привычному циклу агента одну идею: **долгие команды в оболочке можно запускать в фоне**, пока основной поток продолжает вызывать модель и обрабатывать другие инструменты. Ниже — связный разбор: что это за механизм в коде и в документации, как он устроен с точки зрения **инструментов (tools)** и откуда вообще берётся «понимание», что задачу нужно выполнить в отдельном потоке.

Ориентиры в репозитории:

- код: `agents/s08_background_tasks.py`;
- англоязычная выжимка концепции: `docs/en/s08-background-tasks.md`.

---

## Зачем это нужно

Многие команды занимают минуты: `npm install`, `pytest`, `docker build`. Если в цикле агента есть только **блокирующий** запуск shell (как обычный `bash`), то пока команда не закончится, **следующий вызов LLM не происходит** — модель формально «ждёт» вместе с процессом.

Пользователь может сформулировать задачу в духе: «пока ставятся зависимости, создай конфиг». С одним только блокирующим `bash` агент вынужден делать шаги **последовательно**, а не перекрывать ожидание другой работой.

**Решение s08:** команда может быть запущена в **отдельном потоке**; хост **сразу** возвращает модели короткий ответ (например, идентификатор задачи), а когда subprocess завершится, результат **попадает в контекст** перед очередным обращением к API. Так агент может «думать дальше», пока тяжёлая команда крутится параллельно.

Ключевая формула из документации: *fire and forget* — агент не блокируется на время выполнения команды; доставка результата отложена до следующего шага цикла.

## Две «дорожки» для shell: блокирующая и фоновая

В `s08_background_tasks.py` для выполнения команд в файловой системе агента зарегистрированы разные инструменты:

| Инструмент | Что происходит в рантайме |
|------------|---------------------------|
| **`bash`** | Синхронный `subprocess.run` (таймаут до 120 с). Поток цикла агента **ждёт** конца команды. |
| **`background_run`** | Создаётся **`threading.Thread`** (`daemon=True`), внутри потока выполняется тот же `subprocess.run` (таймаут до 300 с). Возвращается **сразу** строка вида «задача `task_id` запущена», без ожидания вывода. |
| **`check_background`** | Чтение состояния из словаря задач: одна задача по `task_id` или список всех. |

Плюс стандартный набор для работы с файлами (`read_file`, `write_file`, `edit_file`) с ограничением путей внутри рабочей директории.

Итого с точки зрения **архитектуры инструментов**: фоновый запуск — это **не отдельная магия уровня «типа скилла»**, а **отдельные tools** с явными именами и описаниями в схеме для API. Модель видит в списке и `bash`, и `background_run`, и выбирает одно из имён при вызове.

**Про «скиллы»:** в терминологии этого курса **skill** (например, s05) обычно про **подгружаемые знания и инструкции**, а не про то, в каком потоке выполняется команда. Здесь же речь именно о **инструментах** — тех же по уровню абстракции, что `read_file` или `bash`, только реализация `background_run` заводит поток и очередь уведомлений.

## Как устроен фон: `BackgroundManager` и очередь

Класс **`BackgroundManager`** держит:

1. **`tasks`** — словарь `task_id → {status, result, command}` для учёта и для `check_background`.
2. **`_notification_queue`** — список уведомлений о **завершении** задач (пишется из фонового потока под `threading.Lock`).
3. Метод **`run(command)`** — регистрирует задачу, стартует поток на **`_execute`**, сразу возвращает текст с `task_id`.
4. **`_execute`** — в потоке запускает shell-команду, обновляет `tasks`, кладёт в очередь краткую запись (в т.ч. урезанный вывод для уведомления).
5. **`drain_notifications()`** — за одну операцию **забирает и очищает** очередь, чтобы каждое завершение обработать ровно один раз.

Цикл агента остаётся **однопоточным** в смысле логики LLM: параллелится только выполнение subprocess’ов в worker-потоках.

## Куда деваются результаты: встраивание в `messages`

Перед каждым вызовом `client.messages.create` в **`agent_loop`** вызывается **`drain_notifications()`**. Если есть накопившиеся уведомления и история диалога уже не пустая, в `messages` добавляется **одна синтетическая реплика пользователя**: блок с тегом `<background-results>`, строки вида `[bg:…] status: …` и короткая пометка, что фоновые задачи завершились. **Отдельное сообщение ассистента после этого не добавляется:** у части моделей Anthropic запрос с **последним сообщением от assistant** даёт ошибку *assistant message prefill* / *conversation must end with a user message*.

Так модель **на следующем шаге** автоматически видит итоги фоновых команд, даже если явно не вызывала `check_background` (инструмент проверки остаётся для ручного опроса «в процессе»).

## Откуда сервис «понимает», что задачу надо вынести в поток?

**Никакого автоматического анализа строки команды** на стороне хоста нет: сервис не смотрит на `npm install` vs `echo hi` и не решает сам переключить режим.

**Понимание — это выбор инструмента моделью:**

1. В запрос к API уходят описания всех tools; в system prompt явно сказано использовать **`background_run`** для долгих операций.
2. Модель возвращает `tool_use` с полем **`name`**: либо `bash`, либо `background_run`, и т.д.
3. Код по **`name`** вызывает соответствующий обработчик: для `background_run` — `BG.run()` (поток), для `bash` — блокирующий subprocess.

Если модель ошибётся и отправит долгую установку через **`bash`**, она так и останется блокирующей. Надёжность паттерна держится на **явном разделении tools + промпт**, а не на эвристике в рантайме.

## Ограничения учебной версии

- Потоки и **GIL** не ускоряют чистый CPU в Python, зато хорошо сочетаются с **ожиданием** внешних процессов.
- **`daemon=True`**: при аварийном завершении процесса Python фоновые задачи не гарантируют «дожить» до конца — для локального REPL это приемлемо.
- Нет распределённой шины событий, ретраев и полноценного жизненного цикла задач как в продакшен-оркестраторах — это сознательное упрощение для одного файла и одной идеи.

## Как потрогать руками

Из корня репозитория (с настроенным `.env`: ключ API и `MODEL_ID`):

```bash
python agents/s08_background_tasks.py
```

Примеры запросов в духе документации:

1. Запустить в фоне что-то вроде `sleep 5 && echo done` и параллельно попросить создать файл.
2. Стартовать несколько фоновых `sleep` с разной длительностью и проверить статусы.
3. Запустить тяжёлый прогон (например, тесты) через `background_run` и продолжить диалог на другие темы, пока в `<background-results>` придёт вывод.

## Краткое резюме

**s08** — это паттерн **отложенной доставки результата долгих команд** в контекст агента через **очередь уведомлений** и **отдельные tools** `background_run` / `check_background`. С точки зрения архитектуры это **инструменты определённой семантики** (фон + опрос), а не отдельный «тип скилла». **Поток** включается не потому, что сервис «понял» команду, а потому что модель вызвала **именно** `background_run` — маршрутизация по имени tool в коде хоста.
