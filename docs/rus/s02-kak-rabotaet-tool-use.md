# Как работает s02: Tool Use

`s02` расширяет `s01`: у агента появляется не один `bash`, а набор более точных инструментов, и модель выбирает между ними по описанию.

---

## Почему `s01` уже не хватает

С одним `bash` агент может сделать почти всё, но это неудобно:
- shell-команды легко ломаются на кавычках и спецсимволах;
- чтение и запись файлов через shell менее надёжны;
- у модели слишком широкий и неструктурированный канал действий.

Поэтому в `s02` вводятся отдельные инструменты для файловых операций.

---

## Что нового в `s02`

```text
User -> LLM -> Tool Dispatch
                |
                +-- bash       -> run_bash
                +-- read_file  -> run_read
                +-- write_file -> run_write
                +-- edit_file  -> run_edit
```

Появляется словарь `TOOL_HANDLERS`, который связывает имя инструмента с конкретной функцией.

---

## 1. Диспетчер инструментов

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

Теперь цикл не вызывает `run_bash(...)` напрямую. Вместо этого он:
- читает `block.name`;
- ищет обработчик по имени;
- вызывает его с `block.input`.

Это и есть dispatch map.

---

## 2. Как модель выбирает нужный инструмент

Модель не видит Python-код напрямую. Она получает только массив `TOOLS`, где для каждого инструмента есть:
- `name`;
- `description`;
- `input_schema`.

Например:

```python
TOOLS = [
    {"name": "bash", "description": "Run a shell command.", ...},
    {"name": "read_file", "description": "Read file contents.", ...},
    {"name": "write_file", "description": "Write content to file.", ...},
    {"name": "edit_file", "description": "Replace exact text in file.", ...},
]
```

По сути модель делает следующее:
1. читает задачу пользователя;
2. смотрит список доступных инструментов;
3. сопоставляет задачу с `description`;
4. формирует аргументы по `input_schema`.

Пример:
- пользователь просит прочитать файл;
- модель выбирает `read_file`;
- в ответе приходит `ToolUseBlock(name="read_file", input={"path": "requirements.txt"})`.

---

## 3. Обновлённый цикл агента

```python
for block in response.content:
    if block.type == "tool_use":
        handler = TOOL_HANDLERS.get(block.name)
        output = handler(**block.input) if handler else f"Unknown tool: {block.name}"
        results.append({
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": output,
        })
```

Сам while-loop не меняется. Меняется только способ выполнения инструмента:
- в `s01` был жёсткий вызов одного `bash`;
- в `s02` появляется маршрутизация по имени инструмента.

---

## 4. Какие инструменты доступны

| Инструмент | Назначение |
| --- | --- |
| `bash` | Выполнение shell-команд |
| `read_file` | Чтение файла, опционально с ограничением по строкам |
| `write_file` | Создание или полная перезапись файла |
| `edit_file` | Точечная замена текста в файле |

Идея в том, чтобы покрыть почти все повседневные coding-задачи четырьмя инструментами, не раздувая cognitive load модели.

---

## 5. Почему описания и схемы так важны

Выбор инструмента держится на качестве `description`.

Если описание нечёткое, модель может выбрать не тот инструмент. Если `input_schema` слишком свободная, возрастает шанс некорректного вызова.

Строгие JSON schema дают два плюса:
- API валидирует аргументы до исполнения;
- намерение модели становится однозначным.

---

## 6. Безопасность путей

Файловые операции идут через `safe_path`, который:
- приводит путь к абсолютному;
- проверяет, что он не выходит за пределы рабочей директории.

Это защищает от путей вроде `../../../etc/passwd`.

---

## Итог

`s02` добавляет два ключевых слоя поверх `s01`:
- структурированные инструменты вместо универсального shell-only подхода;
- карту диспетчеризации, которая направляет вызов на нужный обработчик.

При этом главный цикл агента остаётся тем же: модель решает, какой инструмент вызвать, код исполняет его и возвращает результат обратно.
